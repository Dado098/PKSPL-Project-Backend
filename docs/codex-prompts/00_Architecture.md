## Fixed technical standards

- Laravel 12 API-only backend, PHP 8.3, MySQL 8, Nginx, phpMyAdmin, and Docker Compose.
- API base path: `/api/v1`.
- PSR-12, Laravel Pint, PHPStan/Larastan, PHPUnit feature tests.
- Use Controllers, Form Requests, API Resources, Services, Repository interfaces/implementations, DTOs for service input, Policies, and a dedicated service provider for bindings.
- Keep controllers thin. Business rules belong in services; database queries belong in repositories.
- Use Laravel's dependency injection. Do not use static service locators or a generic base repository.
- Centralize successful API responses and error responses using one project-consistent response helper and Laravel exception handling. Do not return `200` for failed operations.
- Log meaningful application failures and write activity history only for supported actions. Do not manufacture audit data or silently ignore failed logging.
- Use database transactions for multi-write business operations where partial writes would corrupt state.
- Document every public class and non-obvious business rule with concise PHPDoc. Do not add redundant comments to trivial getters or framework boilerplate.
- Do not add packages, database columns, tables, roles, status values, or API fields not required by the SQL schema or an explicit user instruction.

## Database and authorization constraints

The supplied SQL is the source of truth for table names, fields, relationships, and enum values.

1. The SQL declares integer primary keys but does **not** declare `AUTO_INCREMENT`. Before generating migrations, ask the user whether IDs are application-supplied or auto-generated. Do not silently add `id()` / `bigIncrements()`.
2. The schema has `roles` and `users.id_role`, but no `permissions`, `role_permissions`, or `user_permissions` tables. Implement role-based authorization only from the roles that exist; do not claim to implement granular permission management or invent these tables.
3. No table in the SQL declares a deleted-at column. Do not add `softDeletes()` without explicit confirmation.
4. Do not infer which role may perform which action. Define policy methods, but request a role/action matrix before enforcing role-specific rules beyond authentication and ownership checks supported by the schema.
5. Treat all monetary and calculated valuation values as decimal strings at the API boundary; never coerce them to floating-point values.
